# PRD-004 — Autenticação e Controle de Acesso

| Campo | Valor |
|--------|--------|
| **Projeto** | ERP Comercial Grifres |
| **Versão** | 1.0.0 |
| **Status** | Aprovado |
| **Prioridade** | Crítica |

---

# 1. Objetivo

Implementar o sistema completo de autenticação e controle de acesso do ERP Comercial Grifres.

Este módulo será responsável por proteger toda a área administrativa do sistema, permitindo que apenas usuários autenticados tenham acesso às funcionalidades internas.

O login será exclusivo para administradores e colaboradores da empresa.

Não haverá cadastro público de usuários.

---

# 2. Objetivos

- Implementar autenticação segura.
- Implementar autorização baseada em permissões.
- Proteger todas as rotas administrativas.
- Criar estrutura reutilizável para futuros módulos.
- Preparar o sistema para múltiplos níveis de acesso.

---

# 3. Escopo

Este PRD contempla:

- Tela de Login
- Logout
- Recuperação de sessão
- JWT
- Refresh Token
- Middleware de autenticação
- Guards
- Controle de permissões
- Seed do administrador
- Estrutura de usuários

Não contempla:

- Cadastro de usuários
- Gestão de funcionários
- Auditoria
- Alteração de senha pelo usuário
- Recuperação de senha por e-mail

Esses recursos serão implementados futuramente.

---

# 4. Fluxo de Autenticação

```text
Usuário

↓

Tela de Login

↓

Validação

↓

JWT

↓

Refresh Token

↓

Área Administrativa

↓

Logout
```

---

# 5. Tela de Login

A tela deverá conter:

- Logo
- Nome da empresa
- Campo E-mail
- Campo Senha
- Botão Entrar
- Exibir/Ocultar senha
- Loading durante autenticação
- Mensagens de erro

---

# 6. Usuário Administrador

Criar uma seed inicial contendo:

```text
Nome

Administrador

E-mail

admin@grifres.com

Senha

12345678
```

A senha deverá ser armazenada utilizando Bcrypt.

---

# 7. Estrutura do Usuário

```text
id

nome

email

senha

cargo

ativo

ultimoLogin

createdAt

updatedAt
```

---

# 8. Papéis (Roles)

Inicialmente:

```text
Administrador

Gerente

Funcionário
```

A estrutura deverá permitir adicionar novos papéis futuramente.

---

# 9. Permissões

Cada usuário possuirá permissões baseadas no cargo.

Exemplo:

Administrador

- acesso total

Gerente

- gestão operacional

Funcionário

- acesso limitado

As permissões detalhadas serão implementadas conforme os módulos forem desenvolvidos.

---

# 10. JWT

Implementar:

- Access Token
- Refresh Token
- Expiração configurável
- Assinatura segura
- Renovação automática da sessão

---

# 11. Logout

Ao realizar logout:

- remover cookies;
- invalidar sessão;
- redirecionar para Login.

---

# 12. Proteção de Rotas

Toda rota administrativa deverá exigir autenticação.

Exemplos:

```text
/admin

/admin/produtos

/admin/categorias

/admin/pedidos

/admin/clientes

/admin/estoque

/admin/financeiro
```

Rotas públicas:

```text
/

/produtos

/categorias

/contato
```

---

Criar módulo Auth contendo:

```text
Controller

Service

DTO

Repository

Strategy

Guard

Decorator

Interfaces

Tests
```

---

# 14. Frontend

Criar:

```text
Login

Auth Provider

Protected Route

Loading Screen

Session Manager
```

---

# 15. Banco de Dados

Criar tabela de usuários.

Campos mínimos:

```text
id

nome

email

senha

cargo

ativo

ultimo_login

created_at

updated_at
```

---

# 16. Segurança

Obrigatório:

- JWT
- Refresh Token
- Bcrypt
- Cookies HttpOnly
- Validação de DTOs
- Rate Limit
- Helmet
- CORS
- Sanitização de entradas
- Proteção contra SQL Injection
- Proteção contra XSS

---

# 17. Validações

E-mail:

- obrigatório
- formato válido

Senha:

- obrigatória
- mínimo de 8 caracteres

---

# 18. Experiência do Usuário

Durante o login:

- botão desabilitado durante requisição;
- indicador de carregamento;
- mensagens de erro claras;
- redirecionamento automático após autenticação.

---

# 19. Critérios de Aceitação

O PRD será considerado concluído quando:

- Login funcionando.
- Logout funcionando.
- JWT implementado.
- Refresh Token implementado.
- Rotas protegidas.
- Usuário administrador criado via seed.
- Sessão persistente.
- Middleware funcionando.
- Guards funcionando.
- Controle de acesso preparado para expansão.

---

# 20. Fora do Escopo

Não implementar:

- Cadastro de usuários.
- Recuperação de senha.
- Confirmação por e-mail.
- MFA.
- OAuth.
- Login social.
- Gestão de funcionários.
- Auditoria de acessos.

---

# 21. Observações

Este módulo será utilizado por todos os módulos administrativos do ERP Comercial Grifres.

Toda funcionalidade interna deverá obrigatoriamente utilizar este sistema de autenticação e autorização, garantindo segurança, escalabilidade e facilidade de manutenção.
